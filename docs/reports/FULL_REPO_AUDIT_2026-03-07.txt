3. DIRECT ANSWER: IS THE PIPELINE USING REAL DATA OR SYNTHETIC?
============================================================

Short answer:

- Live pipeline: real data
- Historical backtest pipeline: mixed, depending on flags
- Tests/examples/helper scripts: synthetic inputs exist

3.1 Canonical live pipeline uses real data

Evidence:

1. Line collection is from real sportsbook snapshots
   - scripts\collect_lines.py
   - core\nba_line_store.py
   - core\nba_data_collection.py:get_todays_event_props_bulk

   The collector writes append-only JSONL snapshots under data\line_history\.
   Example from data\line_history\2026-03-07.jsonl:
   - real player names
   - real books (FanDuel)
   - real lines and American odds
   - real event ids and commence_time

2. Live projection path uses real player/team history
   - core\nba_prop_engine.py -> compute_prop_ev()
   - core\nba_prep_projection.py -> compute_projection()
   - core\nba_data_collection.py -> NBA API fetches
   - core\nba_data_collection.py:98-110 -> local historical fallback when NBA
     API is unavailable

   Important nuance:
   if the NBA API fails, the fallback is LocalNBAStats (real local historical
   data from the Kaggle index), not synthetic gamelogs.

3. Decision/signal logging uses the observed line context
   - nba_cli\scan_commands.py -> roster_sweep
   - core\gates.py
   - core\nba_decision_journal.py

   roster_sweep reads stored snapshots, computes EV against the snapshot line,
   and journals qualifying signals or blocked leans.

3.2 Backtest pipeline can use synthetic lines

This is explicit in code, not hidden:

- core\nba_backtest.py:268 defines _synthetic_line(proj_stat)
- core\nba_backtest.py:815-817 sets:
  - line = _synthetic_line(proj_stat)
  - over_odds = -110
  - under_odds = -110

When this happens:

- If historical closing line is missing and you are not in --real-only mode
- If odds_source is None
- If you are running broader simulated backtests

How to disable synthetic fallback:

- Use --odds-source local_history --real-only
- Or use match-live mode, which forces real lines only

This separation is well understood in the codebase and docs:
- roiReal = real historical closing lines
- roiSynth = synthetic fallback lines

3.3 Synthetic or shortcut behavior exists outside the canonical live path

Confirmed examples:

1. Tests use synthetic inputs
   - tests\test_compute_ev.py
   - tests\test_gates.py
   - tests\test_betting_policy.py

2. Calibration sample/example artifacts exist
   - examples\calibration_sample.csv
   - models\calibration_sample_model.json
   - models\calibration_sample_model_venv.json

3. Helper script shortcut:
   - scripts\betmgm_scan.py:420
   - falls back to projection = float(line) if stat projection parse fails

   This is not the canonical compute_prop_ev path. It is a research/convenience
   scanner and should not be treated as the source of record.

4. LLM line fallback exists for analysis UX
   - nba_cli\llm_commands.py exposes projectionSource = "line_fallback"
   - this is an explanatory/LLM path, not the main live betting pipeline

Verdict:

- The main live pipeline is using real data.
- The backtest layer can still use synthetic lines by design.
- If you want strict real-only evaluation, use real historical odds modes only
  and ignore synthetic ROI entirely.

4. ARCHITECTURE AUDIT
============================================================

4.1 High-level architecture

The repo is structured more like a real internal tool than a toy project:

- nba_mod.py
  Thin CLI wrapper

- nba_cli/
  Command routing and handlers

- core/
  Actual engine logic

- data/
  Runtime stores, journals, backtests, line history, local indexes

leaking into CLI or UI files.

4.2 Canonical path through the system

CLI/UI entry:
- nba_mod.py
- nba_cli\router.py
- server.py shells nba_mod.py for HTTP endpoints

Live line path:
- scripts\collect_lines.py or nba_mod.py collect_lines
- core\nba_data_collection.py -> Odds API
- core\nba_line_store.py -> data\line_history\YYYY-MM-DD.jsonl

Live analysis path:
- nba_cli\scan_commands.py -> roster_sweep
- core\nba_prop_engine.py -> compute_prop_ev
- core\nba_prep_projection.py -> compute_projection
- core\nba_ev_engine.py -> probability + EV math
- core\gates.py -> qualification rules
- core\nba_decision_journal.py -> signal and lean logging

Historical backtest path:
- nba_cli\backtest_commands.py
- core\nba_backtest.py
- local data via core\nba_local_stats.py or bref via core\nba_bref_data.py or
  NBA API path
- historical odds via core\nba_odds_store.py

4.3 Policy/gate layering

There are two important policy layers:

1. SIGNAL_SPEC in core\gates.py
   Determines whether a signal qualifies

2. BETTING_POLICY in core\nba_data_collection.py
   Determines what stats/bins are in the bettable whitelist

This is intentional and useful:
- reb can still be signal-eligible for research
- but excluded from the actual betting whitelist

Severity: High
------------------------------------------------------------

Two separate problems exist.

Problem A: workflow ordering mismatch

- scripts\nba_agent.py defines daily_scan as:
  collect_lines -> best_today -> roster_sweep

  collect_lines -> roster_sweep -> best_today

The ops_commands order is correct. The nba_agent.py order is stale/wrong.

Impact:

  swept and logged

From data\agent_logs\daily_scan_2026-03-02.log through
data\agent_logs\daily_scan_2026-03-06.log:

- roster_sweep timed out after 120s on every recorded day

This is not a one-off.

Impact:

- automation output can be stale or incomplete
- the daily workflow is not currently reliable end-to-end

Additional note:
- some logs also show "Ollama not running - summary unavailable"
- that is secondary; the primary problem is roster_sweep timing out
